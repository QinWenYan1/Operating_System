<a id="id1"></a>
## ✅ 知识点1: 换出的必要性

**理论**
- `get_free_page()` 不能总是获得新页，**物理内存是有限的**
- 需要**选择一页淘汰**，将其**换出**(`Swap Out`)到磁盘，腾出页框(`Frame`)
- 核心问题：**选择哪一页淘汰？**

**教材示例/公式**
```c
page = get_free_page();
bread_page(page, current->executable->i_dev, nr);
// 当 get_free_page() 失败时，必须换出页面
```

**注意点**
- ⚠️ **关键区分**：换入(`Swap In`)是从磁盘到内存，换出(`Swap Out`)是从内存到磁盘
- 💡 **理解技巧**：就像衣柜满了——买新衣服（换入）前，必须丢掉旧衣服（换出）腾空间
- 🔄 **知识关联**：与 L24 请求调页内存换入直接衔接，换入换出是一对互补操作
- 📋 **术语提醒**：`Swap Out(换出)`、`Frame(页框/帧)`、`Page Replacement(页面置换)`

## ✅ 知识点2: FIFO 页面置换算法

**理论**
- **FIFO**(`First-In First-Out`)：最先进入内存的页面最先被淘汰
- 最容易想到，但性能不佳——刚换入的页可能马上又要使用

**教材示例/公式**
```
页框数：3，引用序列：A B C A B D A D B C B C B A D C B A B C B D A D B A C B A

页框状态：
    3       C C D D D C C C C C B B D D D C C C C D D A A A B B
    2       B B B B B B B B B B A A A A A A A B B B B B B B B A A
    1       A A A A A A A A A A D D D D D D B B A A A A A A C C B

缺页次数：7 次
```

**评价**：**D 换 A 不太合适**——A 刚被使用过，不应该被淘汰！

**注意点**
- ⚠️ **关键问题**：FIFO 可能把刚换入的页马上换出，导致**Belady 异常**（页框增加缺页反而增多）
- 💡 **理解技巧**：FIFO 就像排队买奶茶——先来的先走，不管你现在喝没喝
- 🔄 **知识关联**：与 L24 请求调页直接关联——换入后马上换出是 FIFO 的致命缺陷
- 📋 **术语提醒**：`FIFO(先进先出)`、`Belady's Anomaly(Belady异常)`、`Reference String(引用串)`

<a id="id3"></a>
## ✅ 知识点3: MIN 最优页面置换算法

**理论**
- **MIN** 算法：选择**最远将使用**的页面淘汰，是**理论最优方案**
- 缺点：需要**预知未来**——知道每个页面下次使用的时间

**教材示例/公式**
```
页框数：3，引用序列：A B C A B D A D B C B C D C B A B C B D A D B A C B A

页框状态：
    3       C C C C C C C C C C D D D D D D C C C C D D D D C C
    2       B B B B B B B B B B B B B B B B B B B B B B B B B B
    1       A A A A A A A A A A A A A A A A A A A A A A A A A A

缺页次数：5 次
```

**结论**：MIN 导致 **5 次缺页**，优于 FIFO 的 7 次

**注意点**
- ⚠️ **关键区分**：MIN 是理论最优，但**不可实现**——需要知道未来引用序列
- 💡 **理解技巧**：MIN 就像开天眼——知道未来谁会最久不用，就换谁
- 🔄 **知识关联**：MIN 是评价其他算法的"天花板"——越接近 MIN 越好
- 📋 **术语提醒**：`MIN(Optimal Page Replacement,最优置换)`、`Optimal(最优)`

## ✅ 知识点4: LRU 最近最少使用算法

**理论**
- **LRU**(`Least Recently Used`)：选择**最近最长一段时间没有使用**的页面淘汰
- 核心思想：**用过去的历史预测将来**——最近不用的，将来可能也不用

**教材示例/公式**
```
页框数：3，引用序列：A B C A B D A D B C B C D C B A B C B D A D B A C B A

页框状态：
    3       C C D D D C C C D D D C C C B B B B D D D A A A C C
    2       B B B B B B B B B B B B B B A A A A A A A A B B B B
    1       A A A A A A A A A A A A A A D D D D C C C C B B A A

缺页次数：5 次
```

**结论**：LRU 也导致 **5 次缺页**，与 MIN 一样！

**注意点**
- ⚠️ **关键区分**：LRU 是**近似 MIN** 的实用算法，基于"局部性原理"(`Principle of Locality`)
- 💡 **理解技巧**：LRU 就像整理书架——最久没翻的书，最可能不再需要
- 🔄 **知识关联**：与 L21 内存分区与分页中的程序局部性原理直接关联
- 📋 **术语提醒**：`LRU(Least Recently Used,最近最少使用)`、`Principle of Locality(局部性原理)`

## ✅ 知识点5: LRU 准确实现——时间戳法

**理论**
- 每页维护一个**时间戳**(`Time Stamp`)，记录最近一次访问时间
- 淘汰时选择**时间戳最小**的页

**教材示例/公式**
```
时间戳记录示例：

| 时间 | 操作 | A | B | C | D | 说明 |
|:---|:---|:---|:---|:---|:---|:---|
| 0 | 初始 | 0 | 0 | 0 | - | |
| 1 | 访问A | 1 | 0 | 0 | - | A 时间戳更新为 1 |
| 2 | 访问B | 1 | 2 | 0 | - | B 时间戳更新为 2 |
| 3 | 访问C | 1 | 2 | 3 | - | C 时间戳更新为 3 |
| 4 | 访问A | 4 | 2 | 3 | - | A 时间戳更新为 4 |
| 5 | 访问B | 4 | 5 | 3 | - | B 时间戳更新为 5 |
| 6 | 访问D | 4 | 5 | 3 | 6 | D 时间戳更新为 6；选 A 淘汰！|
```

**注意点**
- ⚠️ **关键问题**：每次地址访问都需要修改时间戳，需维护全局时钟，需找到最小值——**实现代价较大**
- 💡 **理解技巧**：时间戳法就像给每本书贴"最后阅读日期"，找最旧的扔掉——每次都要翻所有书
- 🔄 **知识关联**：准确实现代价高，因此需要近似实现
- 📋 **术语提醒**：`Time Stamp(时间戳)`、`Global Clock(全局时钟)`

## ✅ 知识点6: LRU 准确实现——页码栈法

**理论**
- 维护一个**页码栈**(`Page Stack`)，栈顶是最近访问的页，栈底是最久未访问的页
- 淘汰时选择**栈底页**

**教材示例/公式**
```
页码栈变化示例：

| 访问 | 页码栈（栈顶→栈底） | 操作 |
|:---|:---|:---|
| A | A | 压入 A |
| B | B A | 压入 B |
| C | C B A | 压入 C |
| A | A C B | A 移至栈顶 |
| B | B A C | B 移至栈顶 |
| D | D B A | C 淘汰，压入 D |
| A | A D B | A 移至栈顶 |
| D | D A B | D 移至栈顶 |
| B | B D A | B 移至栈顶 |
| C | C B D | A 淘汰，压入 C |
| B | B C D | B 已在栈顶 |
```

**注意点**
- ⚠️ **关键问题**：每次地址访问都需要修改栈（移动元素，修改约 10 次栈指针）——**实现代价仍然较大**
- 💡 **理解技巧**：页码栈就像把最近看的书放在最上面，最久没看的沉到底——每次都要搬书
- 🔄 **知识关联**：两种准确实现都代价高，因此实际使用近似实现
- 📋 **术语提醒**：`Page Stack(页码栈)`、`Stack Top(栈顶)`、`Stack Bottom(栈底)`

<a id="id7"></a>
## ✅ 知识点7: LRU 近似实现——Clock 算法（SCR）

**理论**
- 每个页加一个**引用位**(`Reference Bit, R`)——硬件自动设置
- 选择淘汰页时：扫描 R 位，**R=1** 时清 **0** 并继续扫描；**R=0** 时淘汰该页
- 组织成**循环队列**，使用一个指针(`hand`)扫描——称为 **Clock 算法**(`Clock Algorithm`)

**教材示例/公式**
```
Clock 算法图示：

        R=1
       ↗   ↘
    R=1       R=0
    ↑           ↓
    R=1       R=1
    ↑           ↓
    R=0       R=1
    ↑           ↓
    R=0 ← R=1 ← R=0

hand 指针顺时针扫描：
- 遇到 R=1 → 清 0，继续扫描
- 遇到 R=0 → 淘汰该页
```

**注意点**
- ⚠️ **关键区分**：SCR(`Second Chance Replacement`) = Clock 算法 = 给第二次机会
- 💡 **理解技巧**：Clock 就像宿舍检查——每间房贴个标签（R=1 表示"刚用过"），检查时把标签撕了（R=0），下次没标签的就搬走
- 🔄 **知识关联**：用硬件引用位降低实现代价，是近似 LRU 的经典方案
- 📋 **术语提醒**：`Clock Algorithm(时钟算法)`、`SCR(Second Chance Replacement,二次机会置换)`、`Reference Bit(引用位)`

## ✅ 知识点8: Clock 算法的改进——双指针时钟

**理论**
- **Clock 的问题**：如果缺页很少，所有页的 R=1，`hand` 扫描一圈后回到起点，**退化为 FIFO**！
- **原因**：记录了太长的历史信息——很久前的访问都保留为 R=1

**改进方案：双指针时钟**
- **快指针**：用来**清除 R 位**，移动速度**快**——定期"刷新"历史记录
- **慢指针**：用来**选择淘汰页**，移动速度**慢**——真正淘汰页面

**教材示例/公式**
```
改进后的 Clock 算法：

        R=1
       ↗   ↘
    R=1       R=0
    ↑           ↓
    R=1       R=1
    ↑           ↓
    R=0       R=1
    ↑           ↓
    R=0 ← R=1 ← R=0

- 快指针（清 R）：快速转圈，把所有 R=1 清成 R=0
- 慢指针（淘汰）：转得慢，遇到 R=0 就淘汰

更像真正的 Clock 了！
```

**注意点**
- ⚠️ **关键区分**：快指针清 R 相当于"忘记久远历史"，慢指针淘汰基于"近期未使用"
- 💡 **理解技巧**：快指针就像定期清理——把旧标签都撕了；慢指针就像真正淘汰——没标签的搬走
- 🔄 **知识关联**：与 L22 多级页表和快表中的 TLB 刷新机制有类似思想
- 📋 **术语提醒**：`Fast Hand(快指针)`、`Slow Hand(慢指针)`、`Double-Hand Clock(双指针时钟)`

<a id="id9"></a>
## ✅ 知识点9: 页框分配与颠簸现象

**理论**
- 需要给每个进程分配**页框**(`Frame`)数量
- 分配太多 → 请求调页的高效利用失去意义
- 分配太少 → **颠簸**(`Thrashing`)现象

**教材示例/公式**
```
颠簸现象：

多道程序程度
    ↑
    │                    ╱
CPU │                 ╱
利  │              ╱
用  │           ╱
率  │        ╱  ← 急剧下降（颠簸区域）
    │     ╱
    │  ╱
    └────────────────────→ 进程个数

- 进程增多 → 每个进程的页框减少 → 缺页率增大
- 缺页率增大到一定程度 → 进程总等待调页 → CPU 利用率降低
- 进程进一步增多 → 缺页率更大 → 恶性循环
```

**注意点**
- ⚠️ **关键区分**：颠簸是**缺页过多导致系统忙于换页，CPU 利用率急剧下降**的现象
- 💡 **理解技巧**：颠簸就像教室座位不够——人太多， constantly 站起来换座位，根本没时间听课
- 🔄 **知识关联**：与 L14 CPU 调度策略中的多道程序设计程度直接关联
- 📋 **术语提醒**：`Thrashing(颠簸)`、`Frame Allocation(页框分配)`、`Degree of Multiprogramming(多道程序程度)`

## ✅ 知识点10: 完整页换入换出流程

**理论**
- 完整的内存管理流程：磁盘 → 页表 → 物理内存 → 换入换出
- `swap 分区`用于存放换出的页面
- `swap out` 时调用 `bwrite` 写磁盘；`swap in` 时从 swap 分区读取

**教材示例/公式**
```
┌─────────┐     ┌─────────┐     ┌─────────┐
│  磁盘   │     │  页表   │     │ 物理内存 │
└────┬────┘     └────┬────┘     └────┬────┘
     │               │               │
     │               │    load [addr]│
     │               │◄──────────────┘
     │               │       i (缺页中断)
     │               │    do_no_page
     │               │       ↓
     │               │    访问缺页
     │               │       ↓
     │◄──────────────┴──── bread ────┐
     │        swap分区               │
     │               ↑               │
     │               │    get_free_page
     │               │               ↓
     │               │◄───────── put_page
     │               │               │
     │               │    重新执行    │
     │               │◄──────────────┘
     │               │
     │    ┌─────────┴─────────┐
     │    │                   │
     │  swap out           swap in
     │    │                   ↑
     └────┼───────────────────┘
          │    swap分区管理
          │
     ┌────┴────┐
     │ bwrite  │  ← swap out：将页面写到 swap 分区
     │         │
     │ swap in │  ← swap分区管理：读取换入页面
     └─────────┘
```

**注意点**
- ⚠️ **关键区分**：`swap 分区`是磁盘上专门用于页面换入换出的区域，不同于普通文件系统
- 💡 **理解技巧**：swap 分区就像"临时仓库"——搬出去的东西暂时放这里，需要时再搬回来
- 🔄 **知识关联**：与 L24 请求调页内存换入的流程图直接衔接，形成完整闭环
- 📋 **术语提醒**：`Swap Partition(swap分区)`、`bwrite(块写)`、`swap in/out(换入换出)`

## 🔑 核心要点总结

1. **换出的必要性**：物理内存有限，`get_free_page()` 失败时必须淘汰页面换出到磁盘
2. **页面置换算法对比**：FIFO(7次缺页) → MIN(5次，理论最优) → LRU(5次，近似最优)
3. **LRU 实现代价**：时间戳法、页码栈法准确但代价大，实际使用近似实现
4. **Clock 算法**：引用位 R=1 清 0 继续扫描，R=0 淘汰；缺页少时退化为 FIFO
5. **双指针改进**：快指针清 R 位，慢指针淘汰，避免记录太长历史
6. **颠簸现象**：页框分配太少导致缺页率过高，CPU 利用率急剧下降
7. **完整流程**：缺页 → 找空闲页（无则换出）→ 从磁盘读入 → 更新页表 → 继续执行

## 📌 考试速记版

- **关键机制**：`get_free_page()` 失败 → 选择淘汰页 → `bwrite` 换出到 swap 分区 → 换入新页
- **算法对比**：

| 算法 | 核心思想 | 缺页次数 | 可实现性 |
|:---|:---|:---|:---|
| FIFO | 先进先出 | 7次 | ✓ 易实现 |
| MIN | 最远使用 | 5次 | ✗ 需预知未来 |
| LRU | 最近最少使用 | 5次 | △ 近似实现 |
| Clock | 二次机会 | ≈5次 | ✓ 高效实现 |

- **Clock 算法**：R=1 → 清 0，继续；R=0 → 淘汰。双指针：快指针清 R，慢指针淘汰
- **颠簸**：缺页过多导致系统忙于换页，CPU 利用率急剧下降

**易混淆概念对比**：

| 概念 | 换入(Swap In) | 换出(Swap Out) |
|:---|:---|:---|
| 方向 | 磁盘 → 内存 | 内存 → 磁盘 |
| 触发 | 缺页中断 | 内存不足 |
| 函数 | `bread_page` | `bwrite` |
| 位置 | swap 分区 → 物理页 | 物理页 → swap 分区 |

**常见考试陷阱**：
- Clock 算法缺页少时**所有 R=1**，扫描一圈后**退化为 FIFO**！
- LRU 是**近似 MIN**，不是等于 MIN——只是在这个例子中结果相同
- 时间戳法和页码栈法都是**LRU 的准确实现**，但**代价大，实际不用**
- **颠簸**是**缺页率过高**导致的 CPU 利用率下降，不是进程太多本身

**记忆口诀**：
> 换入换出一对拳，内存有限必须换；FIFO 太傻先淘汰，MIN 最优看未来；LRU 用历史近似它，时间戳和栈代价大；Clock 指针转圈圈，R=1 清 0 R=0 搬；双指针快慢分开转，太久历史不算账；页框太少要颠簸，CPU 利用率掉深渊！
