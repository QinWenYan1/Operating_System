# 第1章 操作系统基础 (Operating System Fundamentals)

> **章节定位**：操作系统课程的入门章节，从计算机硬件结构出发，建立操作系统全貌认知——涵盖操作系统定义与定位、计算机启动流程（从图灵机到冯诺依曼结构）、x86 PC开机引导与操作系统启动（setup/head模块、保护模式切换）、操作系统接口与系统调用实现、操作系统发展历史，以及后续CPU/内存/文件管理三大主线的学习路线图。

## 📋 章节导航与重难点

### 1.1 [什么是操作系统](./1.1_What_is_OS.md)
- **核心**：
  - `操作系统(Operating System)` 的定义与定位——位于硬件与应用之间的系统软件层，提供**方便性**和**高效性**两大价值
  - `裸机(Bare Machine)` 概念与直接使用硬件的局限——直接操作显存地址（如 `mov 0x68 → 777`）复杂且低效
  - 操作系统管理的八大硬件资源——CPU、内存、终端、磁盘、文件、网络、电源、多核
  - 操作系统学习的三个层次——用接口（`printf`/`fork`/`open`）→ 理解实现 → 设计实现
- **难点**：
  - 裸机编程 vs 有操作系统编程的对比理解——从直接写硬件地址到调用统一接口的本质转变
  - 操作系统学习层次的递进关系——从"会用"到"懂原理"到"能动手改"的跨越

### 1.2 [揭开钢琴的盖子](./1.2_Open_the_OS.md)
- **核心**：
  - `图灵机(Turing Machine)` 与 `通用图灵机(Universal Turing Machine)`——从专用计算到可编程计算的演进
  - `冯·诺依曼存储程序思想`——存储程序 + 顺序执行，计算机五大部件结构
  - x86 PC 开机时 CPU 状态——`CS=0xFFFF, IP=0x0000` → 物理地址 `0xFFFF0`（ROM BIOS），实模式寻址
  - `引导扇区(Boot Sector)`——硬盘 0 磁道 0 扇区的 512 字节，BIOS 加载到 `0x7c00`，末尾必须是 `0xAA55`
  - `bootsect.s` 引导代码三部曲——自我移动（`0x7c00→0x9000`）→ 加载 setup → 加载 system → 跳转到 setup
- **难点**：
  - 从图灵机到冯诺依曼的演进逻辑——通用图灵机的"菜谱"概念如何演变为存储程序思想
  - 实模式寻址原理——`CS:IP = CS×16 + IP`，开机地址 `0xFFFF0` 的推导
  - 引导扇区加载机制——BIOS 为什么加载到 `0x7c00`，引导代码为什么要自我移动

### 1.3 [操作系统启动](./1.3_OS_Boot.md)
- **核心**：
  - 启动三阶段——`bootsect`（引导）→ `setup`（实模式初始化+进保护模式）→ `head`（保护模式初始化+进main）→ `main`（C语言初始化）
  - `setup` 模块——获取硬件参数、设置 GDT/IDT、开启 A20 地址线、置 `cr0.PE=1` 进入保护模式，最后 `jmpi 0,8` 跳转到 `head.s`
  - 保护模式地址翻译——实模式直接 `cs×16+ip`，保护模式通过 `GDT` 查表转换
  - `head.s` 的 32 位初始化——重新设置 GDT/IDT、开启分页、设置栈、最终通过 `ret` 巧妙跳转到 `main()`
  - `mem_init` 内存管理初始化——`mem_map[]` 数组标记每个物理页的状态
- **难点**：
  - 保护模式地址翻译机制——GDT 表项结构（8字节 = 基址32位 + 限长20位 + 属性12位）与段选择子的解析
  - A20 地址线的历史遗留问题——为什么通过 8042 键盘控制器开启，解决 1MB 地址回绕问题
  - 控制流的巧妙跳转——`setup_paging` 后的 `ret` 通过栈操作直接跳到 `main()`，而不是返回到调用处

### 1.4 [操作系统接口](./1.4_OS_Interface.md)
- **核心**：
  - 操作系统接口的本质——`系统调用(System Call)`，表现为 C 语言函数调用，是用户访问内核的唯一合法途径
  - 接口的核心价值——**屏蔽细节**，连接应用软件和内核，使用者不需要知道内部实现
  - 命令行和图形界面的本质——都是**外壳(Shell)**，是用户态程序，最终都要通过系统调用访问内核
  - `Shell` 的核心机制——`fork()` 创建子进程 + `exec()` 执行程序 + `wait()` 等待返回
  - `GUI` 基于消息驱动——硬件输入 → 系统消息队列 → 消息循环 → 窗口过程函数处理
  - `POSIX` 标准——定义进程管理、文件系统等核心接口，保证应用程序的可移植性
- **难点**：
  - 接口"屏蔽细节"的本质理解——从电源插座、汽车油门到系统调用，接口的核心作用是连接和隔离
  - 系统调用 vs 库函数的区别——系统调用直接进入内核，库函数在用户态执行，可能只封装系统调用

### 1.5 [系统调用的实现](./1.5_System_Call_Implementation.md)
- **核心**：
  - 内存隔离是安全基础——用户程序不能直接访问内核内存，必须借助硬件机制实现隔离
  - `CPL(Current Privilege Level)` / `DPL(Descriptor Privilege Level)` / `RPL(Request Privilege Level)`——特权级检查由 x86 处理器硬件完成
  - `int 0x80` 是进入内核的唯一合法途径——用户态通过中断门切换到内核态，中断门 DPL=3 允许用户触发
  - 系统调用 = 中断 + 查表 + 参数传递——`eax` 存调用号，`ebx/ecx/edx` 存参数，查 `sys_call_table` 调用内核函数
  - 库函数封装——用户不直接写 `int` 指令，通过 C 库（如 `_syscall3` 宏）间接调用，确保安全性
- **难点**：
  - 特权级检查硬件机制——`DPL ≥ CPL` 且 `DPL ≥ RPL` 的条件判断，以及为什么中断门 DPL=3 是关键
  - CPL vs DPL vs RPL 的区分——CPL 是当前执行的特权级，DPL 是目标段的特权级，RPL 是请求特权级
  - 为什么系统调用必须通过中断——不能简单地在内存中放一个函数指针让用户调用

### 1.6 [操作系统历史](./1.6_History_of_Operating_Systems.md)
- **核心**：
  - 操作系统历史两条主线——大型机线（IBSYS→OS/360→MULTICS→UNIX→Linux）追求资源利用率，PC线（CP/M→MS-DOS→Windows/Mac）追求用户体验
  - `批处理操作系统`——1955-1965，IBM 7094，最大化硬件利用率，单道顺序执行
  - `多道程序操作系统`——OS/360，多道并发，CPU 调度，资源利用率提升
  - `分时系统`——MULTICS，多用户轮转，每个用户感觉独占计算机
  - `UNIX` 的诞生——1969年，简化 MULTICS，影响后续所有系统（Linux、Mac OS、iOS）
  - `Linux` 的诞生——1991年，Linus Torvalds，开源内核，推动现代操作系统发展
- **难点**：
  - 历史两条主线的演进逻辑——大型机线从"批处理→多道→分时→多用户多任务"，PC线从"单任务→图形界面→多任务"
  - 各系统的继承关系——UNIX 是分水岭，Linux 是 UNIX 的开源实现，Mac OS 底层是 BSD UNIX
  - 关键时间线与因果关系——先有分组交换思想，后有 TCP/IP，再有 Web 应用爆发

### 1.7 [回顾和计划](./1.7_Recap_and_Plan.md)
- **核心**：
  - 操作系统管理的四大硬件资源——CPU、内存、磁盘、终端设备，通过抽象层向用户提供简单接口
  - 四大核心抽象——CPU → `进程(Process)`（`fork()`）、内存 → `虚拟地址(Virtual Address)`（`*p = 7`）、磁盘/设备 → `文件(File)`（`open/read/write`）
  - 后续课程三条主线——CPU 管理（进程与多进程切换）、内存管理（虚拟内存与地址映射）、文件/设备管理（文件抽象与设备驱动）
  - 系统调用是入口，内核机制是本质——用户通过简单接口触发，内核完成复杂的资源调度和管理
- **难点**：
  - 抽象与硬件的映射关系——进程不是真实硬件，虚拟地址不是物理地址，文件不是磁盘块，理解抽象的本质
