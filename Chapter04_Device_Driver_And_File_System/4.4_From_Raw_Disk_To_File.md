## ✅ 知识点1: 为什么引入文件——生磁盘的问题

**理论**
- **生磁盘(`raw disk`)**的访问方式要求用户直接操作盘块号（扇区号），这对普通用户极不友好
- 许多用户连**扇区(`sector`)**是什么都不知道，更不可能根据盘块号来访问磁盘
- 需要在盘块之上引入更高层次的抽象概念——**文件(`file`)**
- 文件是操作系统对磁盘使用的**第三层抽象**：
  - 第一层：生磁盘 → 通过盘块号直接读写
  - 第二层：磁盘驱动 → 使用`CHS`或`LBA`寻址
  - 第三层：**文件系统** → 用户通过文件名访问，无需关心物理位置

**注意点**
- ⚠️ **关键区分**：生磁盘(raw disk) vs 熟磁盘(cooked disk)——文件系统让磁盘变得"对用户友好"
- 💡 **理解技巧**：文件本质是"用户视角的字符流"与"磁盘视角的盘块集合"之间的桥梁
- 📋 **术语提醒**：`生磁盘(raw disk)`、`熟磁盘(cooked disk)`、`盘块(disk block)`

## ✅ 知识点2: 文件的本质——字符流到盘块集合的映射

**理论**
- **用户眼里的文件**：一个**字符序列(`character stream`)**，即字符流
- **磁盘上的文件**：一个**盘块集合(`block collection`)**
- **文件的核心作用**：建立**字符流到盘块集合的映射关系(`mapping`)**
- 用户操作的是字符位置（如第200-212个字符），文件系统负责将其映射到对应的物理盘块

**注意点**
- ⚠️ **关键区分**：用户视角（字符流）vs 磁盘视角（盘块集合）——这是文件系统的核心设计问题
- 🔄 **知识关联**：与L28生磁盘的使用对比——L28是直接操作盘块，L29是通过文件间接操作
- 📋 **术语提醒**：`字符流(character stream)`、`盘块集合(block collection)`、`映射(mapping)`

<a id="id3"></a>
## ✅ 知识点3: 连续结构实现文件

**理论**
- **连续结构(`contiguous allocation`)**：文件在磁盘上占用连续的盘块
- **映射方式**：通过文件名找到起始块号和块数，即可计算任意字符对应的盘块位置
- **示例**：
  - 文件`test.c`：起始块=6，块数=3
  - 占用盘块：6, 7, 8
  - 若盘块大小为100字节，则第200-212字符对应：
    - 第200字节 → 盘块 $6 + \lfloor 200/100 \rfloor = 8$ 中的偏移 $200 \bmod 100 = 0$
    - 即盘块8中的第0-12字节
- **文件控制块(`FCB, File Control Block`)**：存储文件名、起始块、块数等元数据

**注意点**
- ⚠️ **关键区分**：连续结构的优点是访问速度快（直接计算），缺点是**文件长度增减困难**（可能涉及大量数据移动）
- 💡 **理解技巧**：像数组一样随机访问，但插入/删除代价高
- 🔄 **知识关联**：类似内存分配中的连续分配（如分区分配），存在外部碎片问题
- 📋 **术语提醒**：`FCB(File Control Block)`、`连续结构(contiguous allocation)`、`起始块(starting block)`

## ✅ 知识点4: 链式结构实现文件

**理论**
- **链式结构(`linked allocation`)**：文件盘块通过指针链接，每个盘块存储下一个盘块的地址
- **映射方式**：`FCB`只存储文件名和**起始块号**，通过依次跟随指针找到后续盘块
- **示例**：
  - 文件`test.c`：起始块=1
  - 盘块1 → 指针指向盘块9 → 指针指向盘块10 → 指针指向-1（表示结束）
- **优点**：文件长度增减容易（只需修改指针）
- **缺点**：
  - **顺序访问慢**（必须依次读取，无法随机访问）
  - **可靠性差**（一个指针损坏，后续数据全部丢失）

**注意点**
- ⚠️ **关键区分**：链式结构牺牲随机访问能力，换取动态扩展的灵活性
- 💡 **理解技巧**：像链表一样，插入/删除容易，但访问第N个元素必须遍历
- 🔄 **知识关联**：与内存管理中的链表分配类似
- 📋 **术语提醒**：`链式结构(linked allocation)`、`指针(pointer)`

## ✅ 知识点5: 索引结构实现文件

**理论**
- **索引结构(`indexed allocation`)**：连续和链式分配的有效折衷，实际操作系统中较常用
- **映射方式**：
  - `FCB`存储文件名和**索引块号**
  - 索引块中存储文件所有盘块号的列表
- **示例**：
  - 文件`test.c`：索引块=19
  - 索引块19的内容：`[9, 17, 1, 10, -1, -1, -1, ...]`
  - 文件占用盘块：9, 17, 1, 10（-1表示空闲/结束）
- **访问第200-212字符**（假设块大小100）：
  - 先读取索引块19，找到盘块列表
  - 第200字节 → 盘块 $\lfloor 200/100 \rfloor = 2$ 号列表项 → 盘块1，偏移0
  - 即盘块1中的第0-12字节

**注意点**
- ⚠️ **关键区分**：索引结构需要一次额外的磁盘读取（读索引块），但支持随机访问且扩展灵活
- 💡 **理解技巧**：像数组的数组（二维索引），先查"目录"再找"内容"
- 🔄 **知识关联**：与内存管理中的页表思想类似——用额外空间换随机访问能力
- 📋 **术语提醒**：`索引结构(indexed allocation)`、`索引块(index block)`

## ✅ 知识点6: 多级索引与inode

**理论**
- **UNIX `inode`(`index node`)**：实际系统采用的多级索引结构
- **inode结构**：
  - 直接指针：12个`ptr[1..12]`，直接指向数据块
  - 一阶间接指针：`ptr[13]`指向一个索引块，该块含128个数据块指针
  - 二阶间接指针：`ptr[14]`指向一个索引块，该块含128个一阶间接索引块指针
  - 三阶间接指针：`ptr[15]`指向一个索引块，该块含128个二阶间接索引块指针
- **最大文件大小计算**（假设块大小=1KB，指针大小=4B）：
  - 直接指针：$12 \times 1KB = 12KB$
  - 一阶间接：$128 \times 1KB = 128KB$
  - 二阶间接：$128 \times 128 \times 1KB = 16MB$
  - 三阶间接：$128 \times 128 \times 128 \times 1KB = 2GB$
  - 总计：约 **2GB+**
- **设计优势**：
  - **很小的文件高效访问**：12个直接指针覆盖大多数小文件
  - **中等大小的文件访问速度也不慢**：只需一次间接索引
  - **很大的文件也能表示**：多级索引支持极大文件

**注意点**
- ⚠️ **关键区分**：直接指针 vs 间接指针——小文件直接用直接指针，大文件才用间接指针，避免小文件访问额外开销
- 💡 **理解技巧**：inode像一棵倒立的树，根是直接指针，叶是数据块，中间节点是索引块
- 🔄 **知识关联**：与多级页表、多级 cache 思想一致——用分级结构平衡"小文件效率"和"大文件容量"
- 📋 **术语提醒**：`inode(index node)`、`直接指针(direct pointer)`、`间接指针(indirect pointer)`、`一阶间接(single indirect)`、`二阶间接(double indirect)`、`三阶间接(triple indirect)`

<a id="id7"></a>
## ✅ 知识点7: 文件存储结构的选择

**理论**
- **不同场景选择不同的文件结构**：
  - **连续结构**：适合**只读不写的静态文件**（如词典、光盘文件系统），访问最快
  - **链式结构**：适合**频繁增删的日志型文件**，但现代系统很少直接使用
  - **索引结构**：**通用操作系统的首选**，兼顾随机访问和动态扩展
- **思考题**：专门为像词霸这样的词典设计文件存储，采用哪种结构最好？
  - **答案：A. 顺序（连续结构）**
  - 原因：词典文件通常是**只读**的，大小固定，需要最快的随机访问速度

**注意点**
- ⚠️ **关键区分**：没有"最好的结构"，只有"最适合场景的结构"——通用系统选索引，专用系统可选连续
- 💡 **理解技巧**："顺序读词典"——词典是静态数据，连续结构最合适
- 🔄 **知识关联**：与数据库索引、文件系统设计等后续内容关联
- 📋 **术语提醒**：`连续结构(contiguous)`、`链式结构(linked)`、`索引结构(indexed)`

## 🔑 核心要点总结

1. **文件是磁盘的第三层抽象**：让用户通过文件名操作字符流，隐藏物理盘块细节
2. **三种实现结构**：连续（快但难扩展）、链式（易扩展但慢）、索引（折衷方案）
3. **inode多级索引**：小文件直接访问，大文件间接索引，兼顾效率与容量
4. **通用操作系统选索引结构**：UNIX/Linux的inode是经典设计，体现操作系统设计的魅力
