## ✅ 知识点1: 终端设备组成与外设工作机制

**再次回到最初的起点...**
- CPU 与外设交互遵循标准的4步流程：
    1. **CPU 写命令到控制器寄存器**：CPU 向**控制器**下达指令（写入控制寄存器）
    2. **控制器驱动外设执行**：控制器翻译命令，驱动**外设硬件**（如显卡芯片、键盘电路）独立工作，不占用 CPU
    3. **控制器向 CPU 发中断**：外设任务完成后，由**控制器**通知 CPU 操作完成
    4. **读数据到内存**：数据从控制器/外设搬运到内存（可由 DMA 完成）
    ![alt text](images/2.png)

- **三层关系**：CPU ↔ **控制器（含寄存器）** ↔ **外设硬件**
  - **外设**：物理设备本身（显卡、键盘）
  - **控制器**：外设内的芯片/电路，负责协议翻译和时序控制
  - **寄存器**：控制器内部的存储单元，是 CPU 与外设交互的**唯一接口**

- **核心要点**：
    1. 通过 out 指令向外设发命令，out 是 x86 CPU 直接给外设的控制寄存器发输出的指令
    2. 系统将外设抽象为件，根据文件描述结构中的信息(inode) 决定执行具体什么 out 指令
    3. 中断处理

## ✅ 知识点2: 键盘中断初始化与中断入口

**键盘的故事应该从哪里开始？ -- 中断处理！**
- **两个视角对比**

    | 视角 | 行为 | 机制 |
    |:---|:---|:---|
    | 使用者（人） | 敲键盘、看结果 | 物理交互 |
    | 操作系统 | "等着"你敲键盘，敲了就中断 | **中断驱动**(`Interrupt-Driven`) |

- **中断初始化代码**

    ```c
    void con_init(void)  // 键盘也是 console 的一部分
    {
        set_trap_gate(0x21, &keyboard_interrupt);  // 设置 0x21 号中断门
    }
    ```

- **键盘中断处理入口**（`kernel/chr_drv/keyboard.S`）

    ```asm
    .globl _keyboard_interrupt
    _keyboard_interrupt:
        inb $0x60,%al           // 从端口 0x60 读扫描码(Scan Code)
        call key_table(,%eax,4) // 调用 key_table+eax*4（查表散转）
        ...
        push $0
        call _do_tty_interrupt  // 调用上层处理
    ```
- **主要任务**：
    1. `inb` 是 从 `I/O` 端口读 1 字节 的指令
    2. 从键盘端口 0x60 读取扫描码，通过查表找到对应处理函数，最后调用上层 `_do_tty_interrupt` 完成输入处理。



> 📋 **术语提醒**：`扫描码`(`Scan Code`)是键盘硬件输出的按键编码，不是 ASCII 码！需要映射转换

<a id="id3"></a>
## ✅ 知识点3: 扫描码与 `key_table` 函数数组

**接下来又是查对应处理函数...**
- **`key_table` 函数数组**（`kernel/chr_drv/keyboard.S`）

    ```asm
    key_table:
        .long none,do_self,do_self,do_self   // 扫描码 00-03
        .long do_self, ...,func, scroll, cursor 等等
    ```

- **主要任务**：`key_table` 是一个**函数指针数组**，通过 `key_table + eax * 4` 实现**散转**（跳转）
- **关键函数**：`do_self` —— 显示字符通常都用此函数处理
> 🔄 **知识关联**：这与 `crw_table` 的设计思想完全一致——都是用函数指针数组实现"查表跳转"

- **扫描码对应关系**
    - 扫描码 `01` → 按键 `ESC`
    - 扫描码 `02` → 按键 `1`
    - 扫描码 `12` → 按键 `E`
    - 等等


> ⚠️ **关键区分**：扫描码(`Scan Code`) ≠ ASCII 码！扫描码是硬件按键编号，ASCII 是软件字符编码
> 💡 **理解技巧**：`call key_table(,%eax,4)` 是 x86 的间接调用——基址 `key_table` + 索引 `eax` × 4（每个函数指针 4 字节），实现多路分支

<a id="id4"></a>
## ✅ 知识点4: 键盘映射表机制

**那么 `do_self` 是干啥的**
- **`do_self` 函数**：
    ```asm
    mode: .byte 0

    do_self:
        lea alt_map,%ebx
        testb $0x20,mode        // alt 键是否同时按下？
        jne 1f
        lea shift_map,%ebx
        testb $0x03,mode        // shift 键是否同时按下？
        jne 1f
        lea key_map,%ebx        // 默认使用 key_map
    1:                          // 找到映射表
    ```
- **主要任务**：处理普通字符，根据修饰键选择映射表
    >⚠️ **关键区分**：`mode` 字节记录当前修饰键状态（Shift、Alt、Ctrl、Caps Lock），`testb` 指令按位检测

- **映射表选择逻辑**

    | 条件 | 使用的映射表 | 示例 |
    |:---|:---|:---|
    | Alt 按下 | `alt_map` | Alt 组合键 |
    | Shift / Caps Lock | `shift_map` | `a` → `A` |
    | 默认 | `key_map` | `a` → `a` |

    >💡 **理解技巧**：三个映射表 = 三套键盘布局，运行时根据 `mode` 选择——这是键盘"本地化"的核心机制
    >📋 **术语提醒**：`key_map` 是默认映射表，`shift_map` 是 Shift 按下时的映射，`alt_map` 是 Alt 组合键映射

- **键盘映射表定义（条件编译）**

    美国键盘（`KBD_US`）：
    ```asm
    key_map:
        .byte 0,27
        .ascii "1234567890-="   // 扫描码对应的 ASCII 字符
        ...

    shift_map:
        .byte 0,27
        .ascii "!@#$%^&*()_+"   // Shift 后的映射
        ...
    ```

<a id="id5"></a>
## ✅ 知识点5: `do_self` 函数与 ASCII 码转换

**选择好了映射表之后...**
- **`do_self` 续**：完成扫描码到 ASCII 码的转换

    ```asm
    1:  movb (%ebx,%eax),%al    // 扫描码索引，ASCII 码 → al
        orb %al,%al
        je none                 // 没有对应的 ASCII 码，返回

        testb $0x4c,mode        // 看 caps 是否亮
        je 2f
        cmpb $'a,%al
        jb 2f
        cmpb $'},%al
        ja 2f
        subb $32,%al            // 小写变大写（ASCII 差 32）

    2:  testb $??,mode          // 处理其他模式，如 ctrl 同时按下

    3:  andl $0xff,%eax
        call put_queue
    none:
        ret
    ```
- **主要任务**：
    1. `movb (%ebx,%eax),%al`：用扫描码 `eax` 索引映射表 `ebx`，取输入的 ASCII 码 
    2. `call put_queue`：将 ASCII 码放入输入队列 

> ⚠️ **关键区分**：`subb $32,%al` 是小写转大写的核心——ASCII 中 `'a'(97) - 32 = 'A'(65)`，所有字母连续排列所以统一减 32

## ✅ 知识点6: `put_queue` 与 `read_q` 队列

**如何放入到输入队列中 ...**
- **`put_queue`**

    ```asm
    put_queue:
        movl _table_list,%edx
        movl head(%edx),%ecx
    1:  movb %al,buf(%edx,%ecx)
    ```
- **主要任务**：
    1. `movl head(%edx),%ecx`：得到 `read_q` 的 `head`，并将 ASCII 码放入队列

- **`read_q`在哪里？ -- 在`tty_queue` 结构**

    ```c
    struct tty_queue *table_list[] = {
        &tty_table[0].read_q,    // 读队列
        &tty_table[0].write_q,   // 写队列
        ...
    };
    ```

- **当前进度**：`con.read_q` 已有数据（键盘输入放入读队列）

- **之后**：就是和其他文件接上如 `scanf` 将 `read_q` 挪动到 `write_q`，然后通过 4.1 内容继续后续动作
- **目前还差什么？**： **回显**

## ✅ 知识点7: `do_tty_interrupt` 与 `copy_to_cooked`

**回显怎么做？**
- **上层处理：`do_tty_interrupt`**

    ```c
    void do_tty_interrupt(int tty)  // 上面传来的是 0
    {
        copy_to_cooked(tty_table + tty);
    }
    ```

- **`copy_to_cooked`：核心处理（加工原始输入）**

    ```c
    void copy_to_cooked(struct tty_struct *tty)
    {
        GETCH(tty->read_q, c);          // 从读队列取字符

        if (L_ECHO(tty)) {              // 回显判断，也可以不回显
            PUTCH(c, tty->write_q);     // 放入写队列
            tty->write(tty);            // 立刻显示到屏幕上
        }

        PUTCH(c, tty->secondary);       // 完成 copy_to_cooked
        ...
        wake_up(&tty->secondary.proc_list);  // 唤醒等待进程
    }
    ```
- **主要任务**：
    1. 从终端读队列取出字符，若开启回显则放入写队列并立刻刷到屏幕，再把处理后的字符放入 secondary 队列。

    2. 最后唤醒正在等待该终端输入的进程，让其可以读取处理后的字符。

> ⚠️ **关键区分**：`copy_to_cooked` 是"加工"层——原始键盘输入（`read_q`）经过处理后变成"熟"输入（`secondary`），供上层程序读取
> 💡 **理解技巧**：`cooked` 意为"加工过的"——包括回显、特殊字符处理（如回车换行转换）、行缓冲等

## ✅ 知识点8: 回显机制与三级队列结构

**理论**
- **回显控制**：`L_ECHO(tty)` 标志决定是否回显
  - 若开启：字符同时放入 `write_q` 并调用 `tty->write()` 显示到屏幕
  - 若关闭：字符只进入 `secondary`，不回显（如密码输入）

- **三级队列结构**

| 队列 | 作用 | 方向 |
|:---|:---|:---|
| `read_q` | 原始键盘输入（扫描码转换后）| 硬件 → 内核 |
| `write_q` | 回显输出队列 | 内核 → 屏幕 |
| `secondary` | 加工后的输入队列（供程序读取）| 内核 → 进程 |

**数据流向**
```
键盘硬件 ──→ keyboard.S ──→ read_q ──→ copy_to_cooked ──→ secondary
                                               ↓
                                          write_q ──→ con_write ──→ 屏幕
```

**注意点**
- ⚠️ **关键区分**：`secondary` 是**加工后队列**，`read_q` 是**原始队列**——`copy_to_cooked` 负责从"生"到"熟"的转换
- 💡 **理解技巧**：三级队列实现了"输入-加工-输出"的流水线解耦：键盘只管放（`read_q`），TTY 层负责加工（`copy_to_cooked`），程序从 `secondary` 取
- 📋 **术语提醒**：`L_ECHO` 是 TTY 线路规程(`Line Discipline`)的标志位之一，控制终端行为

## ✅ 知识点9: 键盘处理完整流程

**理论**
- **完整处理流程（FSM 列表形式）**

```
- **状态1: 空闲等待**
  - 事件: `无按键`
  - 动作: `CPU 执行其他进程` → `继续等待`
  - 下一状态: `空闲等待`

- **状态2: 键盘中断触发**
  - 事件: `按键按下`
  - 动作: `硬件发出 IRQ1 中断` → `CPU 进入 keyboard_interrupt`
  - 下一状态: `扫描码读取`

- **状态3: 扫描码读取**
  - 事件: `inb $0x60,%al`
  - 动作: `从 0x60 端口读取扫描码` → `call key_table(,%eax,4)`
  - 下一状态: `映射处理`

- **状态4: 映射处理**
  - 事件: `key_table 散转`
  - 动作: `do_self: 选择 key_map/shift_map/alt_map` → `movb (%ebx,%eax),%al`
  - 下一状态: `ASCII 加工`

- **状态5: ASCII 加工**
  - 事件: `获得 ASCII 码`
  - 动作: `Caps/Shift/Ctrl 处理` → `call put_queue`
  - 下一状态: `队列缓冲`

- **状态6: 队列缓冲**
  - 事件: `put_queue 完成`
  - 动作: `字符放入 read_q` → `call do_tty_interrupt`
  - 下一状态: ` cooked 加工`

- **状态7: cooked 加工**
  - 事件: `copy_to_cooked 调用`
  - 动作: `GETCH(read_q, c)` → `if L_ECHO: PUTCH(write_q)` → `tty->write()` → `PUTCH(secondary, c)` → `wake_up()`
  - 下一状态: `空闲等待`
```

**注意点**
- ⚠️ **关键区分**：整个流程中，**中断处理程序（keyboard.S）** 必须快速完成——只做读取和入队，复杂处理交给上层的 `copy_to_cooked`
- 💡 **理解技巧**："中断快、延后慢"——这是设备驱动设计的黄金法则，避免中断处理过长导致丢键

## ✅ 知识点10: 完整设备驱动架构与数据流

**理论**
- **系统调用层次结构**

| 层级 | 文件/模块 | 功能 |
|:---|:---|:---|
| 系统调用 | `read_write.c` | `write` 系统调用入口 |
| 字符设备接口 | `char_dev.c` | `crw_table[]` 字符设备表 |
| TTY 设备写 | `tty_io.c` | `tty_write` |
| TTY 设备读 | `tty_io.c` | `tty_read` |
| 显示器写 | `console.c` | `con_write` |
| 键盘处理 | `keyboard.S` | 键盘中断、扫描码处理 |
| 回显 | `tty_io.c` | 从 `secondary` 反馈 |

- **完整数据流**

```
keyboard.S ──→ read_q ──┐
                         ├──→ copy_to_cooked ──→ secondary ──→ tty_read
                         │              ↓
                         │           write_q ──→ con_write (回显)
                         │
write 系统调用 ──→ tty_write ─────────────────────────────────→ 显示器
```

**思考题分析**

| 问题 | 修改位置 | 影响范围 |
|:---|:---|:---|
| F12 让 `ls` 输出全变 `*` | 修改输出显示（`con_write` 或字符设备层）| **所有输出字符** |
| 只让回显为 `*` | 修改 `copy_to_cooked` 中的回显路径 | 仅回显，不影响实际输出 |
| 输出到文件的内容变 `*` | 修改文件系统层或 `tty_write` 处理 | 实际写入文件的内容 |

**注意点**
- ⚠️ **关键区分**：思考题揭示了三层修改的不同影响面——**回显层**只影响屏幕显示，**设备层**影响所有输出，**文件系统层**影响持久化内容
- 💡 **理解技巧**："回显是镜子，write 是出口"——回显（`write_q`）只是给用户看的反馈，真正的数据流向由系统调用路径决定
- 🔄 **知识关联**：这是 L26（显示器输出）和 L27（键盘输入）的完整闭环——键盘输入经加工后可触发输出回显，形成人机交互循环

## 🔑 核心要点总结

1. **键盘中断驱动**：键盘不是轮询设备，而是**中断驱动**——按键触发 0x21 号中断，`keyboard_interrupt` 从 0x60 端口读取扫描码
2. **扫描码 → ASCII 的两级映射**：`key_table` 函数数组散转 → `key_map/shift_map/alt_map` 查表 → `do_self` 完成 ASCII 转换和修饰键处理
3. **三级队列解耦**：`read_q`（原始输入）→ `copy_to_cooked` 加工 → `secondary`（供程序读取）+ `write_q`（回显输出）
4. **回显可控**：`L_ECHO(tty)` 决定键盘输入是否回显到屏幕——密码输入场景会关闭回显
5. **中断快、延后慢**：`keyboard.S` 只做最小工作（读端口+入队），复杂处理交给 `copy_to_cooked`，避免丢键
